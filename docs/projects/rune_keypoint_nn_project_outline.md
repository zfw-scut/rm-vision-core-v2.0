# 神符轮廓角点神经网络项目讨论大纲

日期：2026-06-18

文档类型：项目路线图 / 新需求讨论大纲

## 1. 项目背景

当前神符识别库主要依赖颜色二值化、轮廓提取、几何规则、PnP 解算、滤波和追踪完成识别。其中已激活扇叶的角点提取规则较复杂，涉及轮廓方向序列、线段匹配和突起点筛选。

后续希望引入一个专门用于轮廓角点检测的神经网络模块，用于提升复杂光照、光晕、拖影、局部遮挡和轮廓形变场景下的角点稳定性。

当前阶段仍处于方案讨论期，本文只记录需求共识、讨论顺序和待确认问题，不提前规定具体实现文件、接口、推理后端或训练工程结构。

## 2. 当前共识

- 不用网络替代现有全图二值化主路径。
- 不做端到端姿态回归。
- 网络优先作用在候选轮廓点集上，而不是整张图像上。
- 现有几何校验、PnP、滤波和追踪仍然是最终输出链路的重要组成部分。
- 网络应更像角点候选器，而不是最终裁判。
- 首个讨论对象聚焦在已激活扇叶角点，暂不扩大到全部神符特征。
- 实时性是核心约束，任何网络方案都需要证明新增耗时可接受。

## 3. 当前第一阶段

当前阶段先完成弱标注数据闭环，不直接进入模型设计。

已知素材情况：

- 有效视频素材共 7 个。
- 单个视频长度约 1 分钟。
- 当前识别参数不一定适用于全部视频，可能需要逐视频微调颜色、阈值或相关识别参数。

第一阶段目标：

- 使用现有神符识别算法处理视频帧。
- 记录每帧是否正常识别到目标。
- 记录候选轮廓、候选特征、已激活扇叶角点、耗时和失败阶段。
- 将当前帧、轮廓点集和当前规则算法输出角点关联起来，形成后续训练样本的弱标签来源。
- 通过结构化日志帮助 Codex 判断哪些视频、帧段或参数配置更可靠。
- 保留少量带识别绘制结果的图片样例，方便 Codex 后续直接观察识别问题。
- 从结构化日志中直接导出弱标签候选样本，候选样本暂只要求识别成功、存在已激活扇叶、角点数为 6、轮廓点集存在。
- 后续再生成带角点绘制的审查视频，供人工暂停记录异常帧或异常区间。

第一阶段数据筛选原则：

- 数据集划分应按视频划分，避免相邻帧泄漏导致验证结果虚高。
- 第一版样本数量可以不大，但应优先保证弱标签稳定。
- 对参数敏感的视频，应保留参数配置与输出结果的对应关系。
- 多组参数下输出一致的帧可以视为更高置信样本。
- 掉帧预测得到的结果不作为第一版角点弱标签。

## 4. 首轮关注对象

第一阶段讨论对象暂定为已激活扇叶轮廓角点。

候选角点语义暂按 6 点讨论：

```text
top_left
top_center
top_right
side_left
side_right
bottom_center
```

这一选择的原因：

- 已激活扇叶当前手写规则最复杂。
- 该部分对光晕、拖影和遮挡较敏感。
- 角点质量会直接影响后续 PnP 和重投影效果。
- 相比全图检测，轮廓角点网络更轻量，更符合实时性要求。

该范围仍可调整。是否扩展到残缺扇叶、未激活靶心缺口或未激活灯臂，需在第一阶段实验后再讨论。

## 5. 讨论顺序

### 5.1 任务定义

需要先明确网络到底回答什么问题。

待讨论内容：

- 网络输入是轮廓坐标、轮廓方向序列，还是二者组合。
- 网络输出是角点坐标、轮廓序列索引、热力分布，还是多种形式组合。
- 角点是否必须落在轮廓上。
- 遮挡时如何表达角点不可见。
- 角点语义如何在不同观察方向下保持一致。

### 5.2 数据来源

需要明确训练样本从哪里来，以及是否能覆盖真实困难场景。

待讨论内容：

- 使用实车视频、离线测试视频，还是仿真/合成数据补充。
- 当前几何算法的输出能否作为初始弱标签。
- 哪些样本必须人工复核。
- 是否需要单独收集光晕、过曝、拖影、遮挡和侧视样本。
- 样本如何划分训练集、验证集和真实测试集。

### 5.3 标注方式

需要先确定标注粒度和质量要求，再决定工具或格式。

待讨论内容：

- 标注图像角点，还是标注轮廓序列上的点。
- 可见性标记如何定义。
- 残缺轮廓是否参与第一阶段标注。
- 人工标注和规则弱标签如何合并。
- 标注误差允许范围是多少。

### 5.4 模型路线

需要比较不同模型形式的收益和代价。

待讨论内容：

- 1D 序列模型是否足够。
- 是否需要引入局部图像 patch 辅助。
- 坐标回归和序列热力图哪种更适合当前任务。
- 模型大小和推理耗时如何约束。
- 是否需要同时输出置信度或角点可见性。

### 5.5 评估方式

需要先定义“好”的标准，避免只看训练损失。

待讨论内容：

- 单点像素误差如何统计。
- 多大误差可以认为角点可用。
- 是否以 PnP 可解率作为关键指标。
- 是否以重投影误差作为最终指标。
- 如何评估连续帧稳定性。
- 如何和现有规则方法进行公平对比。

### 5.6 集成策略

等模型路线基本明确后，再讨论如何接入核心库。

待讨论内容：

- 网络输出与现有角点提取逻辑如何衔接。
- 网络失败时是否回退规则方法。
- 哪些几何校验必须保留。
- 运行时是否允许关闭网络。
- 如何统计网络新增耗时。

本阶段暂不规定具体 C++ 类名、文件名、目录层级或推理后端。

## 7. 主要风险

- 数据质量不足，导致模型只是学习当前规则的错误输出。
- 轮廓输入规范不稳定，训练和部署分布不一致。
- 标注角点语义不清，导致左右、上下或遮挡点混乱。
- 网络推理耗时超过收益。
- 网络角点偶发跳变，影响 PnP 和 tracker 稳定性。
- 实验只在干净样本上有效，对真实困难场景提升有限。
- 只有 7 个有效视频，随机按帧划分会让验证集和训练集过于相似。
- 逐视频参数差异可能使弱标签混入参数偏置。

## 8. 下一步讨论重点

下一轮讨论应优先回答：

- 如何批量跑完 7 个视频并对比不同参数配置。
- 哪些字段可以作为弱标签置信度筛选依据。
- 人工审查视频需要显示哪些内容。
- 第一批可用于训练的帧如何从诊断结果中导出。
- 第一版是否只做已激活扇叶 6 点。
